# Documentation

## Docstrings

- All public modules, classes, and functions must have docstrings
- Use **Google style**: summary line, then `Args:`, `Returns:`, `Raises:`, `Example:` sections
- Put types in function signatures, not repeated in docstrings
- Docstrings describe purpose and behavior, not types

## Class & Module Docstrings

- Class docstrings: describe purpose + list `Attributes:`
- Module docstrings: describe purpose + list key functions/classes

## Comments

- Explain *why*, not *what*
- Code should be self-documenting for *what*
- Use comments only for non-obvious business logic or workarounds
