# F-1 — Sync to exchange clock (`adjustForTimeDifference`)

- **Fix ID:** F-1
- **Version:** 0.1.0 (shipped in the first release)
- **Date:** 2026-06-20
- **Commit:** `dd19521`
- **Related phase:** 11 (surfaced while live-testing authenticated calls)

## Symptom

All authenticated Bybit calls fail with `retCode 10002` (server-timestamp /
`recv_window`) once the local clock drifts past ~5 s. `status`, `orders`,
`cancel`, `performance`, and `rebalance` all exit `3` (`EXCHANGE_ERROR`).

## Root cause

`ExchangeStore._build_client` constructed the ccxt client without
`options.adjustForTimeDifference`, so every signed request used the unsynced
local clock. When local time and the exchange clock differ by more than the
exchange's `recv_window`, the venue rejects the signature as stale.

## Fix

Set `options.adjustForTimeDifference = True` on the ccxt client. ccxt then
fetches the server-time offset during `load_markets` — which every unified
private call invokes — and applies it when signing, so requests stay valid
regardless of local clock skew. This is the canonical ccxt remedy (see
`docs/cctx/02-exchanges.md`).
