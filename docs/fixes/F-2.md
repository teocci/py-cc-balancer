# F-2 — `auth login` honors the testnet precedence, not just the default

- **Fix ID:** F-2
- **Version:** unreleased (post-0.1.0)
- **Date:** 2026-06-21
- **Commit:** `1fddd07`
- **Related phase:** Auth

## Symptom

`auth login` saves the profile, but the credential check fails for valid
**mainnet** API keys: Bybit returns `retCode 10003 "API key is invalid."` and
the command exits `3` (`EXCHANGE_ERROR`). `auth status` then reports
`valid: null`. The same keys work fine on mainnet — the only workaround was to
pass an explicit `--no-testnet`.

## Root cause

`_cmd_auth_login` resolved the venue with:

```python
testnet = c.DEFAULT_TESTNET if args.testnet is None else args.testnet
```

`DEFAULT_TESTNET` is `True`, and this line ignored both `CCB_TESTNET` and the
TOML `[global] testnet` setting — unlike every other command, which resolves the
flag through `config._resolve_testnet` (precedence: flag > env > TOML > default).
So whenever `--testnet/--no-testnet` was omitted, a new profile always landed on
the sandbox and was verified against the testnet, where mainnet keys are
rejected.

## Fix

Add `config.resolve_login_testnet(override, cli_config)` applying the app-wide
precedence (explicit flag > `CCB_TESTNET` env > TOML `[global] testnet` >
default), and call it from `_cmd_auth_login`. No profile is consulted — the
profile being created is what defines its own venue. The safety default
(testnet) is unchanged when nothing is configured, so a bare `auth login` still
defaults to the sandbox; users reach mainnet with `--no-testnet` or
`CCB_TESTNET=false`.

## Files changed

| File | Change |
|---|---|
| `src/ccbalancer/config.py` | new `resolve_login_testnet()` (in `__all__`); reuses `_resolve_testnet` with no profile |
| `src/ccbalancer/cli.py` | `_cmd_auth_login` calls `config_mod.resolve_login_testnet(args.testnet, args.config)`; corrected the login-parser comment |
| `tests/test_cli_auth.py` | three tests: default-sandbox, honors `CCB_TESTNET`, `--no-testnet` overrides env |
